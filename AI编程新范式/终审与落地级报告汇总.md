### F-02：DeviceSimulator payload 字段双重转义，压测数据全部畸形

| 属性 | 值 |
|------|-----|
| 来源阶段 | 阶段 4（全链路联通） |
| 严重度 | **P0 — 数据质量严重损坏** |
| 涉及文件 | `DeviceSimulator/Infrastructure/HttpTelemetrySender.cs` 第 29 行, `DeviceSimulator/Infrastructure/MqttTelemetrySender.cs` 第 57 行 |

**现象**：Simulator 构建请求体时 `payload = envelope.PayloadJson`，`PayloadJson` 是 `string` 类型（如 `"{\"value\":42}"`）。序列化后发送的 JSON 为 `"payload":"{\"value\":42}"`（字符串），而非 `"payload":{"value":42}`（对象）。Gateway 的 `TelemetryRequest.Payload` 是 `JsonElement?` 类型，会将其解析为 `JsonValueKind.String`。随后 `TelemetryEnvelopeMapper` 再 `JsonSerializer.Serialize()` 这个 string 类型的 JsonElement，导致 Kafka 和 PG 中的 `payload_json` 被双重引号转义，无法作为正常 JSON 使用。

### F-03：MQTT 连接成功后 broker 断连，Gateway 永不重连

| 属性 | 值 |
|------|-----|
| 来源阶段 | 阶段 3（重灾区排查修正验证） |
| 严重度 | **P0 — MQTT 通道永久失效** |
| 涉及文件 | `IoTGateway/Infrastructure/Mqtt/MqttIngestionService.cs` 第 70-72 行 |

**现象**：MQTT 连接成功后进入 `await Task.Delay(Timeout.Infinite, stoppingToken)`。此 delay 仅在 `stoppingToken` 取消时才被唤醒。若 MQTT broker 宕机导致连接断开，`DisconnectedAsync` 回调只记录日志但不会触发 delay 取消，主循环因 `break`（第 72 行）永远不会重新创建 client。MQTT 通道从此静默死亡，仅 HTTP 通道存活。健康检查会报 MQTT unhealthy，但 readiness 探针可能导致 Pod 被摘流量。

**✅ 落实状态**：仓库 `IoTGateway/Infrastructure/Mqtt/MqttIngestionService.cs` 已采用 `TaskCompletionSource` + `Task.WhenAny(..., Task.Delay(Timeout.Infinite, stoppingToken))` 与指数退避；《第 2 步开发计划》§1.9 中 `MqttIngestionService.cs` 全文已与仓库对齐。第二部分「修正 F-03」内嵌的示例为早期草案，**以实现代码及 §1.9 为准**。

| 属性 | 值 |
|------|-----|
| 来源阶段 | 阶段 2（架构审查） |
| 严重度 | **P0 — K8s 部署后可用性归零** |

**现象**：当前只有一个 `/health/ready` 端点同时检查 Kafka + MQTT。若 K8s deployment 将此端点同时配置为 liveness 和 readiness 探针（目前 deploy yaml 存在此倾向），当 Kafka 短暂不可用时，所有 Gateway Pod 同时被 kill 并重启。重启后 Kafka Producer 需要获取 PID（`EnableIdempotence=true`），大量 Pod 并发重启加剧 Kafka 压力，形成正反馈死亡螺旋。

**现象**：Gateway 使用自己的 `SerializerSetup.TightOptions` 序列化 `TelemetryEnvelope` 写入 Kafka；BackendProcessor 使用 `IoTHunter.Shared.Infrastructure.SerializerSetup.TightOptions` 反序列化。两个类独立维护，若任一方修改 `PropertyNamingPolicy` 或 `PropertyNameCaseInsensitive` 而另一方未同步，反序列化将静默返回所有字段为 default 值的 envelope（不抛异常），导致 PG 写入空数据、Redis 投影异常。

### F-07：PersistenceWorker 批量失败后不跳过已处理消息，无限重试死循环

| 属性 | 值 |
|------|-----|
| 来源阶段 | 阶段 4（代码细节） |
| 严重度 | **P1 — 消费者卡死** |
| 涉及文件 | `BackendProcessor/Infrastructure/Kafka/TelemetryPersistenceWorker.cs` 第 155-162 行 |

**现象**：`ProcessBatch` 失败时执行 `throw`（第 161 行），异常被 `ConsumeLoop` 的 `catch (OperationCanceledException)` 外的隐式传播捕获——但实际上外层没有 catch 非 OperationCanceledException 的分支。`Task.Run` 的未观察异常将根据 `BackgroundServiceExceptionBehavior.Ignore` 被忽略，但 offset 未提交。Worker 停止消费，积累 lag 直到重启。若批量中某条消息持续导致 PG 写入失败（如字段过长），每次重启都会卡在同一批次。

### 修正 F-03：MQTT 断连后自动重连

**根本原因**：连接成功后 `break` 直接退出 while 循环，且 `Task.Delay(Infinite)` 无法被 MQTT 断连事件唤醒。

**修正文件** — `IoTGateway/Infrastructure/Mqtt/MqttIngestionService.cs`，重写 `ExecuteAsync`：

```csharp
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    _stoppingToken = stoppingToken;
    var factory = new MqttClientFactory();
    var attempt = 0;

    while (!stoppingToken.IsCancellationRequested)
    {
        var client = factory.CreateMqttClient();
        var disconnectedTcs = new TaskCompletionSource(TaskCreationOptions.RunContinuationsAsynchronously);

        client.DisconnectedAsync += e =>
        {
            if (e.ClientWasConnected)
                _logger.LogWarning("MQTT disconnected: {Reason}", e.Reason);
            disconnectedTcs.TrySetResult();
            return Task.CompletedTask;
        };

        client.ApplicationMessageReceivedAsync += OnMessageReceivedAsync;

        try
        {
            await ConnectAndSubscribeAsync(client, stoppingToken);
            _client = client;
            attempt = 0;
            _logger.LogInformation("MQTT ingestion started, TCP={Server}:{Port}",
                _mqttOptions.TcpServer, _mqttOptions.TcpPort);

            // 等待断连或服务停止，二者之一即唤醒
            using var reg = stoppingToken.Register(() => disconnectedTcs.TrySetCanceled());
            try { await disconnectedTcs.Task; }
            catch (TaskCanceledException) { }

            // 清理当前 client
            await DisposeClientAsync(client);
            _client = null;

            if (stoppingToken.IsCancellationRequested) break;

            // 断连后短暂等待再重连（避免 broker 重启瞬间的连接风暴）
            await Task.Delay(_mqttOptions.RetryBaseDelayMs, stoppingToken);
        }
        catch (OperationCanceledException) when (stoppingToken.IsCancellationRequested)
        {
            await DisposeClientAsync(client);
            break;
        }
        catch (Exception ex)
        {
            attempt++;
            var delayMs = _mqttOptions.RetryBaseDelayMs * (1 << Math.Min(attempt - 1, 5));
            _logger.LogError(ex,
                "MQTT connection failed (attempt {Attempt}), retrying in {DelayMs}ms",
                attempt, delayMs);
            await DisposeClientAsync(client);
            try { await Task.Delay(delayMs, stoppingToken); }
            catch (OperationCanceledException) { break; }
        }
    }

    _logger.LogInformation("MQTT ingestion stopping...");
}
```

关键改动：
- 引入 `TaskCompletionSource`，`DisconnectedAsync` 触发 `TrySetResult()` 唤醒主循环
- 连接成功后 `attempt = 0` 重置计数器
- 断连后自然回到 while 循环顶部重新创建 client
- 删除 `break`，改为等待 `disconnectedTcs.Task`

**与当前仓库的差异（不必强行对齐上文代码块）**：落地代码使用 `TaskCompletionSource<bool>`、仅在 `e.ClientWasConnected` 时 `TrySetResult`，并用 `Task.WhenAny(disconnectTcs.Task, Task.Delay(Timeout.Infinite, stoppingToken))` 同时响应断连与优雅停机；连接失败分支仍为指数退避。完整实现见《第 2 步》§1.9 或仓库同名源文件。

### 修正 F-07：PersistenceWorker 添加毒丸消息保护

**根本原因**：`ProcessBatch` 中 `throw` 传播到 `Task.Run`，Worker 静默死亡。批量中若存在持续导致异常的消息，每次重启都卡在同一位置。

**修正文件** — `BackendProcessor/Infrastructure/Kafka/TelemetryPersistenceWorker.cs`，`ConsumeLoop` 方法添加异常容错：

```csharp
private void ConsumeLoop(CancellationToken ct)
{
    var batch = new List<ConsumeResult<Null, string>>();
    var lastFlush = DateTime.UtcNow;
    var consecutiveFailures = 0;

    try
    {
        while (!ct.IsCancellationRequested)
        {
            var result = _consumer.Consume(TimeSpan.FromMilliseconds(500));
            if (result is null)
            {
                if (batch.Count > 0 && (DateTime.UtcNow - lastFlush).TotalSeconds >= 5)
                {
                    FlushWithRetry(batch, ref consecutiveFailures);
                    batch.Clear();
                    lastFlush = DateTime.UtcNow;
                }
                continue;
            }

            var envelope = DeserializeAndExtractTrace(result);
            if (envelope is not null)
                batch.Add(result);

            if (batch.Count >= 100 || (batch.Count > 0 && (DateTime.UtcNow - lastFlush).TotalSeconds >= 5))
            {
                FlushWithRetry(batch, ref consecutiveFailures);
                batch.Clear();
                lastFlush = DateTime.UtcNow;
            }
        }
    }
    catch (OperationCanceledException) { }
    finally
    {
        if (batch.Count > 0) FlushWithRetry(batch, ref consecutiveFailures);
        _consumer.Close();
    }
}

private void FlushWithRetry(List<ConsumeResult<Null, string>> batch, ref int consecutiveFailures)
{
    try
    {
        ProcessBatch(batch);
        consecutiveFailures = 0;
    }
    catch (Exception ex)
    {
        consecutiveFailures++;
        _logger.LogError(ex,
            "Batch failed ({ConsecutiveFailures} consecutive). Skipping {Count} messages to DLQ.",
            consecutiveFailures, batch.Count);

        // 连续失败超过 3 次，将整批发到 DLQ 避免无限卡死
        if (consecutiveFailures >= 3)
        {
            foreach (var r in batch)
                _ = _dlq.ProduceAsync(r.Message.Value, "batch_persistent_failure", CancellationToken.None);

            foreach (var r in batch)
                _consumer.StoreOffset(r);
            _consumer.Commit();

            consecutiveFailures = 0;
            _logger.LogWarning("Skipped {Count} messages to DLQ after {Max} consecutive batch failures", batch.Count, 3);
        }
    }
}
```
