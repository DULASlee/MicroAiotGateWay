## 7.1 子步骤一：OpenTelemetry Collector DaemonSet 部署
**目标**：在每个 K3s 节点部署 OTel Collector 实例，接收业务服务的 OTLP 数据，处理后分发至 Jaeger（Traces）和 Prometheus（Metrics 由业务服务自行暴露，Collector 不采集Metrics，仅转发Traces）。
### 7.1.1 动作项
1. **添加 OpenTelemetry Helm 仓库**
```bash
helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts
helm repo update
```
2. **编写 `deploy/observability/otel-collector-values.yaml`**
```yaml
mode: daemonset
config:
  # 边缘代理架构预留说明：
  # 真实设备流量通过边缘节点（如 Traefik/Nginx）暴露公网，
  # 代理将干净的内网流量转发至网关。网关感知不到代理存在，完全解耦公网脏流量。
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: 0.0.0.0:4317
  processors:
    memory_limiter:
      check_interval: 1s
      limit_mib: 512
      spike_limit_mib: 128
    batch:
      timeout: 5s
      send_batch_size: 512
    probabilistic_sampler:
      sampling_percentage: 5.0
  exporters:
    otlp/jaeger:
      endpoint: jaeger-collector.iothunter.svc:4317
      tls:
        insecure: true
  service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, batch, probabilistic_sampler]
      exporters: [otlp/jaeger]
```
3. **部署**
```bash
helm upgrade --install otel-collector open-telemetry/opentelemetry-collector \
 -n iothunter \
 -f deploy/observability/otel-collector-values.yaml
```
### 7.1.2 验证标准
- `kubectl get daemonset -n iothunter otel-collector` 显示 `DESIRED` = `CURRENT` = `READY`。
- `kubectl logs -n iothunter -l app.kubernetes.io/name=opentelemetry-collector` 显示 `"Everything is ready. Begin running and processing data."`。
- **新增连通性验证**：执行 `kubectl exec -it deploy/iot-gateway -n iothunter -- curl -s http://localhost:4317/v1/traces`，返回包含 `partial success`，证明 Pod 内数据能流入 Collector。
## 7.2 子步骤二：Jaeger 部署
**目标**：部署 Jaeger All-in-One 作为 Trace 后端，接收 Collector 转发的 OTLP 数据，提供查询 UI。
### 7.2.1 动作项
1. **新建 `deploy/observability/jaeger.yaml`**
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: jaeger
  namespace: iothunter
spec:
  replicas: 1
  selector:
    matchLabels:
      app: jaeger
  template:
    metadata:
      labels:
        app: jaeger
    spec:
      containers:
        - name: jaeger
          image: jaegertracing/all-in-one:1.62
          ports:
            - containerPort: 16686
              name: query
            - containerPort: 4317
              name: otlp-grpc
            - containerPort: 4318
              name: otlp-http
          env:
            - name: COLLECTOR_OTLP_ENABLED
              value: "true"
            - name: SPAN_STORAGE_TYPE
              value: "memory"
            - name: MEMORY_MAX_TRACES
              value: "100000"
          # 修补：必须大于 Collector 的 memory_limiter (512Mi)，防止压测时 OOM Kill 导致 Trace 断流
          resources:
            limits:
              cpu: 1
              memory: 2Gi
---
apiVersion: v1
kind: Service
metadata:
  name: jaeger-query
  namespace: iothunter
spec:
  selector:
    app: jaeger
  ports:
    - port: 16686
      targetPort: 16686
      name: query
---
apiVersion: v1
kind: Service
metadata:
  name: jaeger-collector
  namespace: iothunter
spec:
  selector:
    app: jaeger
  ports:
    - port: 4317
      targetPort: 4317
      name: otlp-grpc
```
2. **执行**
```bash
kubectl apply -f deploy/observability/jaeger.yaml
```
### 7.2.2 验证标准
- `kubectl get pods -n iothunter -l app=jaeger` 显示 Running。
- 浏览器访问 `http://localhost:16686`（或 Ingress URL），Jaeger UI 正常打开。
- 左侧 Service 下拉菜单中出现 `IoTGateway`、`BackendProcessor` 等。

## 7.5 子步骤五：Grafana 仪表板导入
**目标**：导入 .NET 运行时仪表板、Kafka Consumer Lag 仪表板和 IoT 网关专属自定义面板（如已制作）。
### 7.5.1 动作项
1. **导入 .NET 通用仪表板（ID: 19924）**
- Grafana 左侧 → Dashboards → Import → 输入 `19924` → 加载，数据源选择 Prometheus。
2. **新建 IoT 网关专属面板与边缘预留面板**
- 进入 Grafana → Dashboards → New → Add Visualization。
- 添加面板：
  - **Gateway QPS by Protocol**：`sum(rate(gateway_requests_total{status="success"}[1m])) by (protocol)`
  - **Kafka Ack Latency P99**：`histogram_quantile(0.99, sum(rate(gateway_kafka_ack_latency_ms_bucket[1m])) by (topic, le))`
  - **Consumer Lag per Group**：若采集了 Lag 指标，使用 `kafka_consumer_group_lag`（后续补充 JMX Exporter 或直接通过 Processor 暴露）。
  - **DLQ Rate**：`sum(rate(dlq_messages_total[1m])) by (reason)`
  - **Redis Latest Projection Updates**：`sum(rate(latest_projection_updated_total[1m])) by (device_id)`（示例维度）
  - **MQTT Message Count by QoS**：`sum(rate(mqtt_messages_total[1m])) by (qos, topic)`
- 保存面板，命名为 `IoTHunter Gateway Overview`。
- **新增空骨架面板（边缘代理占位）**：新建面板 `Edge Gateway Performance (Reserved)`，仅添加标题和“全球节点延迟分布热力图”的占位说明，不写实际 PromQL。

- Grafana 中存在名为 `Edge Gateway Performance (Reserved)` 的空骨架面板。
## 7.6 子步骤六：全链路 Trace 验证与日志关联
**目标**：验证从 HTTP/MQTT → Gateway → Kafka → Processor 的完整 Span 在 Jaeger 中串联，并在 Pod 日志中 grep 一个 `trace_id` 可在 Jaeger 中直接搜索到相同 Trace。
### 7.6.1 动作项
1. **发送测试数据**
```bash
kubectl exec -it deploy/device-simulator -n iothunter -- /bin/sh -c " for i in {1..10}; do curl -s -X POST http://iot-gateway/api/v1/telemetry \ -H 'Content-Type: application/json' \ -d '{\"deviceId\":\"dev-trace-test\",\"metricType\":\"heart_rate\",\"timestamp\":$(date +%s)000,\"sequence\":1}' done"
```
2. **在 Jaeger 中查看 Trace**
- 打开 Jaeger UI。
- 服务下拉选择 `IoTGateway`。
- 操作过滤 `Kafka.Consume telemetry.raw`。
- 点击其中一个 Trace，展开 Span 列表，确认结构为：
  ```
  IoTGateway: HTTP/POST /api/v1/telemetry
  ├── IoTGateway: Kafka.Produce
  └── BackendProcessor: Kafka.Consume telemetry.raw
  └── BackendProcessor: BatchWriteToPostgres
  ```
- 点击任意 Span，查看 Tags，包含 `messaging.system=kafka`、`messaging.destination=telemetry.raw`。
3. **验证日志关联**
```bash
TRACE_ID=$(kubectl logs -n iothunter deployment/iot-gateway --tail=50 | grep -oP 'trace=\K[0-9a-f]{32}' | head -1)
kubectl logs -n iothunter deployment/backend-processor --tail=100 | grep "$TRACE_ID"
```
预期：BackendProcessor 日志中也出现同一个 `trace_id`。
### 7.6.2 验证标准
- Jaeger 中能看到完整的多服务 Span 树，Gateway → Kafka → Processor 全部串联。
- Processor 日志中的 `trace_id` 与 Jaeger 中的 Trace ID 一致。
- 跨协议（HTTP/MQTT）的 Trace 也能在 Jaeger 中找到，MQQT 侧的 Span 名含 `MQTT.Receive`。
