## 8.1 子步骤一：压测准备与基线建立
**目标**：启动 5 副本 Simulator 混合协议压测，开启全量链路采样，配置 Grafana 监控面板。
### 8.1.1 动作项
1. **【终审修复：全量采样模式】**
   压测与链路验证期间，必须将 OTel Collector 采样率临时改为 100%，否则 95% 的 Trace 会被丢弃，导致 Jaeger 验证失败。
   ```bash
   # 临时修改 values.yaml 中的采样率并升级
   sed -i 's/sampling_percentage: 5.0/sampling_percentage: 100.0/' deploy/observability/otel-collector-values.yaml
   helm upgrade otel-collector open-telemetry/opentelemetry-collector -n iothunter -f deploy/observability/otel-collector-values.yaml
   ```
2. **扩容 Simulator 至 5 副本并配置混合协议**
   ```bash
   kubectl scale deployment device-simulator -n iothunter --replicas=5
   # 配置 HTTP 模式（3 副本）
   kubectl set env deployment/device-simulator -n iothunter SIMULATOR__PROTOCOL=http SIMULATOR__DEVICECOUNT=20 SIMULATOR__INTERVALMS=50 SIMULATOR__CRITICALEVENTRATIO=0.03
   # 推荐：创建独立的 mqtt-simulator Deployment（2 副本），避免 env 混乱
   kubectl apply -f deploy/device-simulator-mqtt.yaml
   ```
3. **配置 Grafana 监控面板**
   - 打开 Grafana → 进入 `IoTHunter Gateway Overview` 仪表板。
   - 确认面板实时刷新：Gateway QPS by Protocol、Kafka Ack Latency P99、Consumer Lag per Group、DLQ Rate。
   - 增开浏览器窗口，打开 Jaeger UI。
### 8.1.2 验证标准
- 5 个 Simulator Pod 均为 Running。
- Grafana 面板中 Gateway QPS 曲线开始上升。
- Jaeger 中可立即查询到最近的 Trace（全量采样生效）。
## 8.2 子步骤二：渐进式加压与吞吐量记录
**目标**：逐步提高 Simulator 的发送速率，记录系统在不同 QPS 下的 P99 延迟与 Consumer Lag，找到稳态吞吐上限。
### 8.2.1 动作项
1. **调整 Simulator 参数进行 3 轮加压**
   | 轮次       | DeviceCount | IntervalMs | Concurrency |
   | ---------- | ----------- | ---------- | ----------- |
   | 低负载     | 10          | 200        | 2           |
   | 中负载     | 50          | 50         | 4           |
   | 高负载     | 100         | 10         | 8           |
   | 每轮执行： |             |            |             |
   ```bash
   kubectl set env deployment/device-simulator -n iothunter SIMULATOR__DEVICECOUNT=50 SIMULATOR__INTERVALMS=50 SIMULATOR__CONCURRENCY=4
   kubectl rollout status deployment/device-simulator -n iothunter --watch
   ```
2. **【终审修复：规避幻影稳态】**
   等待 `rollout status` 完成后，**强制额外等待 120 秒**，避开 K8s 滚动重启引起的 1 分钟 Prometheus 滑动窗口波动，确保截取的是绝对稳态数据，再记录 Grafana 指标。
3. **记录核心指标**
   - Gateway QPS（按 protocol）、Kafka Ack Latency P99、Consumer Lag、DLQ Rate、Pod 资源使用率。
### 8.2.2 验证标准
- 至少完成两轮加压，QPS 从 100 稳步提升至 2000+。
- P99 延迟在高负载下 ≤ 目标值（100ms）。
- Consumer Lag 在高负载下未持续增长。
## 8.3 子步骤三：故障注入与韧性验证
**目标**：逐个注入故障，验证系统的自愈和降级能力。
### 8.3.1 动作项
1. **故障一：删除 Kafka Pod**
   ```bash
   kubectl delete pod -n iothunter -l app=kafka
   ```
   **验收标准**：故障期间 Gateway 返回 503，不产生假成功；Kafka 恢复后 30 秒内 Gateway 恢复 202。
2. **故障二：删除 Gateway Pod**
   ```bash
   kubectl delete pod -n iothunter -l app=iot-gateway
   ```
   **验收标准**：滚动重启期间无假成功；Processor 未崩溃，Kafka Lag 在恢复后快速下降。
3. **故障三：删除 Redis Pod**
   ```bash
   kubectl delete pod -n iothunter -l app=redis
   ```
   **验收标准**：Redis 故障不阻塞 PG 写入；恢复后投影追平，Redis 中有最新值。
4. **故障四：时序投影存储故障（若已部署）**
   ```bash
   kubectl delete pod -n iothunter -l app=timescaledb
   ```
   **验收标准**：时序库故障不阻塞主链路，恢复后可追平。
### 8.3.2 验证标准
- 4 类故障注入下，关键数据路径（HTTP → Kafka → PG）未产生数据丢失。
- Grafana 面板和 Jaeger Trace 均能反映异常状态。
## 8.4 子步骤四：数据一致性最终核验
**目标**：逐项确认 4 个存储层的数据一致性、幂等去重效果，以及分级可靠性隔离情况。
### 8.4.1 动作项
1. **Redis 最新值核验**
   ```bash
   kubectl exec -n iothunter deploy/redis -- redis-cli HGETALL device:latest:dev-001
   ```
   预期：`sequence` 为设备 dev-001 的当前最大值。
2. **PostgreSQL 主事实库核验**
   ```bash
   kubectl exec -n iothunter deploy/postgres -- psql -U iotapp -d IoTHunter -c "
   SELECT COUNT(DISTINCT event_id) AS total_unique, COUNT(*) AS total_rows
   FROM telemetry_records WHERE device_id='dev-001';"
   ```
   预期：`total_unique` == `total_rows`（无重复记录）。
3. **【终审修复：关键事件分级可靠性独立核验】**
   验证 Topic 分离策略下的关键事件是否 100% 投递，不被普通遥测掩盖。
   ```bash
   # 查询 Kafka event.critical 的总量
   kubectl exec -n iothunter deploy/kafka -- kafka-run-class kafka.tools.GetOffsetShell --broker-list localhost:9092 --topic event.critical --time -1
   
   # 查询 PG 中标记为关键事件的记录数
   kubectl exec -n iothunter deploy/postgres -- psql -U iotapp -d IoTHunter -t -c "
   SELECT COUNT(DISTINCT event_id) FROM telemetry_records WHERE reliability_level=2;"
   ```
   预期：PG 中 `reliability_level=2` 的记录数与 Kafka `event.critical` 的 Offset 数量完全一致（证明关键事件零丢失）。
4. **Kafka 消息留存核验**
   ```bash
   kubectl exec -n iothunter deploy/kafka -- kafka-run-class kafka.tools.GetOffsetShell --broker-list localhost:9092 --topic telemetry.raw --time -1
   ```
   预期：最新 Offset 接近 Simulator 停止时的发送量。
### 8.4.2 验证标准
- Redis 最新值正确，PG 无重复主记录。
- 关键事件独立核验通过（PG 记录数 == Kafka Offset）。
- 查询 API 返回正确数据。

## 8.6 子步骤六：端到端验收脚本（全自动）
**目标**：一键执行核心验收项，输出 PASS/FAIL 结论。
### 8.6.1 动作项
1. **新建 `scripts/e2e-verify-step8.ps1`**
   ```powershell
   param(
       [int]$DurationSeconds = 60,
       [int]$TargetQPS = 1500
   )
   Write-Host "=== IoTHunter Step 8 E2E Verification ===" -ForegroundColor Cyan
   Write-Host "Target: ≥ $TargetQPS QPS sustained for $DurationSeconds seconds" -ForegroundColor Yellow
   # 1. 健康检查
   Write-Host "[1/6] Health check..." -ForegroundColor DarkYellow
   $gwHealth = kubectl exec -n iothunter deploy/iot-gateway -- curl -s -o /dev/null -w "%{http_code}" http://localhost/health/ready
   if ($gwHealth -ne "200") { Write-Host "FAIL: Gateway not healthy" -ForegroundColor Red; exit 1 }
   Write-Host "OK: Gateway healthy" -ForegroundColor Green
   # 2. 启动压测
   Write-Host "[2/6] Starting pressure test..." -ForegroundColor DarkYellow
   kubectl scale deployment device-simulator -n iothunter --replicas=5
   # 3. 等待稳定
   Write-Host "[3/6] Waiting ${DurationSeconds}s for stabilization..." -ForegroundColor DarkYellow
   Start-Sleep -Seconds $DurationSeconds
   # 4. 抓取 QPS（从 Prometheus 查询）
   Write-Host "[4/6] Fetching QPS from Prometheus..." -ForegroundColor DarkYellow
   $promURL = "http://localhost:9090/api/v1/query?query=sum(rate(gateway_requests_total{status=\"success\"}[1m]))"
   $result = (Invoke-RestMethod -Uri $promURL).data.result
   
   # 【终审修复：判空防崩溃】
   if ($result.Count -eq 0) {
       Write-Host "FAIL: No QPS data in Prometheus (Prometheus may be starting up)" -ForegroundColor Red
       kubectl scale deployment device-simulator -n iothunter --replicas=1
       exit 1
   }
   
   $qps = $result[0].value[1] -as [double]
   Write-Host "Current QPS: $qps" -ForegroundColor Cyan
   # 5. 数据一致性
   Write-Host "[5/6] Checking data consistency..." -ForegroundColor DarkYellow
   $pgCount = kubectl exec -n iothunter deploy/postgres -- psql -U iotapp -d IoTHunter -t -c "SELECT COUNT(DISTINCT event_id) FROM telemetry_records;"
   $pgCount = $pgCount.Trim()
   Write-Host "PG unique events: $pgCount" -ForegroundColor Cyan
   # 6. 结案
   Write-Host "[6/6] Verdict..." -ForegroundColor DarkYellow
   if ($qps -ge $TargetQPS -and [int]$pgCount -gt 0) {
       Write-Host "PASS: QPS ($qps) ≥ Target ($TargetQPS), PG has $pgCount records." -ForegroundColor Green
   } elseif ($qps -lt $TargetQPS) {
       Write-Host "FAIL: QPS ($qps) < Target ($TargetQPS)" -ForegroundColor Red
   } else {
       Write-Host "FAIL: No data in PostgreSQL." -ForegroundColor Red
   }
   
   kubectl scale deployment device-simulator -n iothunter --replicas=1
   ```
### 8.6.2 验证标准
- 脚本执行完毕，输出包含 `PASS`，无 `FAIL`。
