# IoTHunter：智能手环云原生双协议网关 —— 八步严酷执行计划（.NET 10 修正版）

> **版本**：V3.0（2026 .NET 10 生产化修正版）  
> **定位**：构建 **HTTP + MQTT-over-WebSocket 双协议**、具备生产级可靠性边界、安全接入、最终一致性和全链路可观测能力的智能手环云原生数据网关。  
> **原则**：每一步都有可直接运行的完整代码、不可逾越的架构决策（ADR）、可验证的检查点。  
> **可靠性声明**：普通高频遥测采用至少进入 Kafka 后可靠处理；关键事件采用 MQTT QoS 1 / HTTP Kafka Ack 确认。任何进入内存 `Channel` 但未进入 Kafka 的数据，不再宣称零丢失。

## 一、总体架构与核心原则

### 1.1 最终服务架构

| 微服务               | 类型                          | 唯一职责                                                     | 暴露端口                 |
| -------------------- | ----------------------------- | ------------------------------------------------------------ | ------------------------ |
| **IoTGateway**       | ASP.NET Core Web API          | **双协议安全接入网关**<br>• HTTP 接收：`POST /api/v1/telemetry`、`POST /api/v1/events/critical`<br>• MQTT WebSocket 接收：订阅普通遥测和关键事件主题<br>• 完成认证、限流、校验、Trace 注入、Kafka 投递<br>• **HTTP 仅在 Kafka ack 成功后返回 `202 Accepted`；Kafka 不可用返回 `503`**<br>• **绝不直连任何数据库** | 80 / 9464               |
| **BackendProcessor** | ASP.NET Core Web API + Worker | **CQRS 查询 + Kafka 投影处理服务**<br>• `TelemetryPersistenceWorker`：消费 Kafka 写 PostgreSQL/TimescaleDB 主事实库<br>• `LatestValueProjectionWorker`：消费 Kafka 更新 Redis 最新值投影<br>• `TimeseriesProjectionWorker`：可选消费 Kafka 写 InfluxDB/ClickHouse 时序投影<br>• Web API：只读查询最新值、历史明细、时序聚合 | 80 / 9464               |
| **DeviceSimulator**  | .NET 通用主机                 | **双模压测模拟器**<br>支持 `--protocol http` 和 `--protocol mqtt`，区分普通遥测与关键事件，压测 Kafka ack、QoS 0、QoS 1、限流和重试行为。 | 无（或 `/metrics` 端口） |

**关键云原生决策**：
- **零 Consul**：100% K8s DNS + Service 服务发现。
- **网关至纯**：不连接任何数据库，Kafka 是唯一持久化下游和接入确认边界。
- **协议解耦**：HTTP 与 MQTT 共享统一接入服务、校验器、Kafka Producer 和 Trace 传播逻辑；`Channel` 只作为背压缓冲，不作为可靠存储。
- **分级可靠性**：普通遥测可接受 best-effort 或 at-least-once；关键事件必须 Kafka ack 或 MQTT QoS 1 确认。
- **全链路追踪**：使用 OpenTelemetry Propagators 在 Kafka Header 注入/提取 W3C Trace Context。
- **安全与韧性**：设备认证、Topic ACL、TLS、限流、Polly 超时/重试/熔断、NetworkPolicy 最小权限、容器优雅停机。

### 1.2 双协议接入核心抽象

```
HTTP Receiver → Auth → Validate → KafkaProducer → Kafka Ack → 202/503
MQTT Receiver → Auth/ACL → Validate → KafkaProducer → Broker Ack/QoS Policy
```
**设计哲学**：Kafka 是系统的第一可靠边界。协议接入层可以使用 `Channel` 做背压和削峰，但只有 Kafka ack 成功后，系统才承诺接管该数据。

### 1.4 架构决策记录（ADR）—— 最高准则

| 编号        | 决策                | 必须 / 禁止                                                  |
| ----------- | ------------------- | ------------------------------------------------------------ |
| **ADR-001** | 服务发现            | **禁止** Consul；**必须** K8s Service 名称                   |
| **ADR-002** | 网关边界            | **禁止** 引用任何数据库驱动；**必须** 仅写 Kafka             |
| **ADR-003** | Kafka 消费语义      | **禁止** `EnableAutoCommit`；**每个投影 Worker 独立 consumer group，成功处理自身投影后提交 Offset** |
| **ADR-004** | 链路上下文          | **必须** 使用 OpenTelemetry Propagators 在 Kafka Header 注入/提取 W3C Trace Context |
| **ADR-005** | 幂等性              | **必须** 引入 `event_id`，PG 以 `event_id` 唯一约束；Redis 更新必须比较 `recorded_at` 或 `sequence` |
| **ADR-006** | 时序存储            | **优先** 使用 PostgreSQL/TimescaleDB 统一主事实与时序查询；若引入 InfluxDB/ClickHouse，必须作为 Kafka 派生投影 |
| **ADR-007** | 容器发布            | **禁止** `PublishSingleFile` + `linux-musl`；**必须** 常规多文件发布 |
| **ADR-008** | 网络策略            | **必须** 使用支持 NetworkPolicy 的 CNI，例如 Cilium；禁止把 NetworkPolicy 写成不可生效的摆设 |
| **ADR-009** | 双协议抽象          | **必须** 复用统一接入服务、DTO 校验、Kafka Producer、Trace 注入；`Channel` 只用于背压，不承担可靠性 |
| **ADR-010** | MQTT 主题           | 普通遥测使用 `$share/gateway-group/device/+/telemetry`；关键事件使用 `$share/gateway-group/device/+/event/critical` |
| **ADR-011** | 消费者组            | **必须** 分离 `iot-persistence`、`iot-latest-projection`、`iot-timeseries-projection` |
| **ADR-012** | 日志关联            | **必须** 使用自定义 `TraceIdEnricher` 从 `Activity.Current` 提取 TraceId/SpanId |
| **ADR-013** | Prometheus 端口     | **必须** 在 K8s Deployment `ports` 中声明 9464，并添加 Pod 注解 `prometheus.io/port: "9464"` |
| **ADR-014** | MQTT QoS            | 普通遥测允许 QoS 0；关键事件必须 QoS 1。Kafka 只保证进入 Kafka 后的可靠性 |
| **ADR-015** | MQTT Session        | 网关使用稳定 ClientId、持久会话或明确的 session expiry；禁止用 `CleanSession=true` 宣称零丢失 |
| **ADR-016** | Simulator Client ID | 普通压测使用 `simulator-{guid}`；可靠性测试使用稳定 ClientId 验证持久会话 |
| **ADR-017** | Mosquitto 配置      | **必须** 显式声明 WebSocket listener、TLS、认证、ACL、`persistence true` |
| **ADR-018** | PG 写入策略（终决） | **放弃纯 COPY**，**必须** 使用批量 `INSERT ... ON CONFLICT (event_id) DO NOTHING`，消除重试死循环 |
| **ADR-019** | OTel Collector 采样 | 当前采用头采样；关键事件 Trace 可使用更高采样率，尾采样为未来优化项 |
| **ADR-020** | HTTP 接入确认       | HTTP 接口只有在 Kafka produce ack 成功后返回 `202 Accepted`；Kafka 不可用或熔断打开时返回 `503` |
| **ADR-021** | 安全接入            | 禁止匿名生产接入；必须具备 TLS、设备认证、Topic ACL、HTTP 限流和 payload schema 校验 |

### 第 2 步：IoTGateway —— 以 Kafka Ack 为可靠边界的双协议接入网关

**目标**：实现 HTTP 与 MQTT WebSocket 双通道安全接入，通过 Polly 管道写入 Kafka；HTTP 只有在 Kafka ack 成功后才确认接入，MQTT 按普通遥测/关键事件分级可靠。

#### 动作项
1. 定义统一事件模型 `TelemetryEnvelope`：
   - `EventId`：设备侧生成，或网关用 `device_id + sequence + recorded_at + metric_type` 生成。
   - `DeviceId`, `Sequence`, `RecordedAt`, `ReceivedAt`, `MetricType`, `Payload`, `ReliabilityLevel`, `SchemaVersion`。
2. 注册统一接入服务 `TelemetryIngestionService`，负责认证、限流、schema 校验、Trace 注入、Kafka Topic 路由。
3. 可选注册有界 `Channel<TelemetryEnvelope>` 做背压和批处理，但不得把写入 `Channel` 视为接入成功。
4. **HTTP 接收**：
   - `POST /api/v1/telemetry`：普通遥测，校验通过后调用 Kafka Producer。
   - `POST /api/v1/events/critical`：关键事件，必须 Kafka ack 成功。
   - Kafka produce ack 成功返回 `202 Accepted`；超时、熔断、Kafka 不可用返回 `503 Service Unavailable`。
5. **MQTT WebSocket 接收**：
   - 实现 `MqttIngestionService : BackgroundService`。
   - 使用 `MQTTnet` 建立 WebSocket/TLS 连接到 `wss://mosquitto:8083/mqtt`。
   - 普通遥测订阅 `$share/gateway-group/device/+/telemetry`，QoS 0 或 QoS 1 由配置控制。
   - 关键事件订阅 `$share/gateway-group/device/+/event/critical`，必须 QoS 1。
   - 网关使用稳定 ClientId 和持久会话；Broker 开启持久化，避免断连窗口丢关键事件。
6. **Kafka 发送策略**：
   - Topic 分离：`telemetry.raw`、`event.critical`。
   - Producer 使用 `Acks=All`、合理 `MessageTimeoutMs`、幂等 Producer（若客户端支持）。
   - Header 注入 `traceparent`、`event_id`、`schema_version`、`reliability_level`。
   - Polly 只包裹超时、有限重试、熔断；禁止无限重试阻塞接入线程。
7. 健康检查 `/health/ready`：验证 Kafka 连接、Producer 元数据获取、熔断器状态、MQTT Broker 连接。
8. 自定义 Meter：`gateway_requests_total`、`gateway_rejected_total`、`gateway_kafka_ack_latency_ms`、`mqtt_messages_total`（标签 `protocol`, `topic`, `reliability`）。

#### 验证标准
- 启动 Gateway + Mosquitto + Kafka，用 `curl` 发 HTTP 请求，只有 Kafka 中可见消息后才返回 202。
- 用 MQTTX WebSocket/TLS 发送普通遥测和关键事件，Kafka Topic 中分别可见 `telemetry.raw` 与 `event.critical`。
- Kafka Header 含 `traceparent`、`event_id`、`schema_version`。
- 停掉 Kafka，HTTP 请求快速返回 503，`/health/ready` 报告降级；关键 MQTT 消息在 Broker 持久会话恢复后重新投递。

### 第 3 步：BackendProcessor —— Kafka 投影式处理器（幂等、最终一致、防死循环）

**目标**：消费 Kafka，按投影独立写入主事实库、最新值缓存和时序视图。避免把 PG、Redis、InfluxDB 伪装成一个分布式事务，以 Kafka + 幂等写入 + 独立 offset 实现可恢复的最终一致。

#### 动作项
1. **TelemetryPersistenceWorker**（主事实库 Worker）：
   - `EnableAutoCommit = false`，`group.id = "iot-persistence"`。
   - 提取 Header 恢复 Trace，创建子 Span。
   - 消息反序列化后进入批处理器，累积 100 条或 5 秒批量处理。
2. **PG/TimescaleDB 主写入策略**：
   - 构造 SQL：
     ```sql
     INSERT INTO telemetry_records
       (event_id, device_id, sequence, metric_type, payload, recorded_at, received_at, schema_version)
     VALUES (...), (...), ...
     ON CONFLICT (event_id) DO NOTHING;
     ```
   - PG 写入成功后提交当前分区最高连续 Offset。
   - 写入失败不提交 Offset，由 Kafka 重投；重复消息由 `event_id` 安全忽略。
3. **LatestValueProjectionWorker**：
   - `group.id = "iot-latest-projection"`。
   - 消费同一 Kafka Topic，写 Redis `device:latest:{deviceId}`。
   - 更新 Redis 前比较 `recorded_at` 或 `sequence`，禁止旧消息覆盖新值。
   - Redis 投影失败不影响 PG 主存储，只影响最新值查询的短暂新鲜度。
4. **TimeseriesProjectionWorker**：
   - `group.id = "iot-timeseries-projection"`。
   - 优先写 TimescaleDB 连续聚合；如采用 InfluxDB/ClickHouse，则作为派生投影写入。
   - 投影失败进入重试和 DLQ，不阻塞主事实库消费。
5. **DLQ 与重放**：
   - schema 无法解析、非法设备、超出合理时间窗口的数据写入 `telemetry.deadletter`。
   - 提供内部重放工具，按 `event_id` 幂等重放。
6. **查询 API**：
   - `GET /api/v2/devices/{deviceId}/latest` → Redis。
   - `GET /api/v2/devices/{deviceId}/history` → PostgreSQL/TimescaleDB 分页。
   - `GET /api/v2/devices/{deviceId}/timeseries?range=1h` → TimescaleDB/InfluxDB/ClickHouse 投影。
   - 响应中返回 `dataFreshness` 或 `projectionLagMs`，让最终一致性可见。
7. 健康检查 `/health/ready`：探测 Kafka、PG、Redis、时序投影存储，并暴露各 consumer group lag。

#### 验证标准
- 本地全栈启动，双协议发送数据，主事实库、Redis 最新值、时序投影均可查询。
- Jaeger 可见完整链路：HTTP/MQTT → Gateway → Kafka → Processor。
- 模拟 Redis 或时序库故障，PG 主事实库仍可持续写入，对应投影 lag 上升但系统不整体阻塞。
- 模拟 Processor 崩溃重启，观察：无重复主记录，无“写入冲突”死循环错误，Kafka Lag 可恢复至 0。

### 第 4 步：DeviceSimulator —— 双模压测工具

**目标**：构建设备模拟器，支持 HTTP 和 MQTT 两种模式的高并发发送，并能区分普通遥测与关键事件的可靠性策略。

#### 动作项
1. 使用 `Host.CreateDefaultBuilder`，创建 `SimulatorWorker`。
2. 内部架构：单生产者生成随机数据 → `Channel<TelemetryEnvelope>` → N 个消费者 Task 发送。
   - HTTP 模式：`IHttpClientFactory` 调用 `http://iot-gateway/api/v1/telemetry`。
   - MQTT 普通遥测：使用 QoS 0/1 发布到 `device/{deviceId}/telemetry`。
   - MQTT 关键事件：使用 QoS 1 发布到 `device/{deviceId}/event/critical`。
3. 每条消息包含 `event_id`、`sequence`、`recorded_at`，用于端到端幂等和一致性核验。
4. 对 503/超时等瞬态故障进行指数退避有限重试；超过上限计入失败指标。
5. 自定义 Meter：`simulator_sent_total`、`simulator_failed_total`、`simulator_ack_latency_ms`（标签 `protocol`, `topic`, `reliability`, `status`）。
6. 配置：`Protocol`, `Target`, `DeviceCount`, `IntervalMs`, `Concurrency`, `CriticalEventRatio`, `ReliabilityLevel`。

#### 验证标准
- 分别以 `--protocol http` 和 `--protocol mqtt` 启动，Gateway 确认双协议接入。
- Simulator 控制台输出 Kafka ack 成功数、MQTT puback 数、失败重试数，`/metrics`（若开放）显示双协议指标。

### 第 5 步：容器化与边缘适配

**目标**：构建 .NET 10 服务 + Mosquitto 容器镜像，导入 K3s containerd，并移除匿名生产接入。

#### 动作项
1. 编写多阶段 Dockerfile：
   - SDK 阶段使用 .NET 10 SDK。
   - Runtime 阶段使用 ASP.NET Core 10 runtime。
   - 常规多文件发布，禁止 `PublishSingleFile + linux-musl` 组合。
2. Mosquitto 镜像配置：提供完整 `mosquitto.conf`：
   ```
   persistence true
   persistence_location /mosquitto/data/
   
   listener 1883
   protocol mqtt
   
   listener 8083
   protocol websockets
   
   allow_anonymous false
   password_file /mosquitto/config/passwordfile
   acl_file /mosquitto/config/aclfile
   ```
3. 生产环境必须启用 TLS：
   - MQTT 使用 8883。
   - MQTT-over-WebSocket 使用 WSS。
   - 证书和密码文件通过 K8s Secret 注入。
4. 构建镜像并导出 tar，`ctr -n k8s.io images import` 导入 K3s。

#### 验证标准
- .NET 10 镜像体积合理，Mosquitto 镜像可用。
- K3s 节点 `crictl images` 可见所有镜像。
- 未认证 MQTT 客户端连接失败；合法设备只能发布自身 ACL 允许的 Topic。

### 第 6 步：K8s 安全基线部署清单

**目标**：部署包含网络策略、探针、TLS/Secret、MQTT Broker ACL 的完整应用。

#### 动作项
1. 确认 K3s 网络插件为 Cilium 或其他明确支持 NetworkPolicy 的 CNI。
2. 创建 Mosquitto Deployment + Service（暴露 1883 和 8083 端口）。
3. 创建 Secret：
   - Kafka/PG/Redis/时序库连接串。
   - MQTT 用户、密码、ACL、TLS 证书。
   - HTTP 设备鉴权密钥或 JWKS 配置。
4. 部署 NetworkPolicy：
   - Simulator → Gateway / Mosquitto。
   - Gateway → Kafka / Mosquitto。
   - Processor → Kafka / PG / Redis / 时序投影存储。
   - 默认拒绝跨命名空间非必要访问。
5. 部署三服务 Deployment：
   - IoTGateway：`replicas:2`，`terminationGracePeriodSeconds:30`，`ports` 含 80 和 9464，Pod 注解 `prometheus.io/port: "9464"`，环境变量注入 `MqttBrokerWebSocketUrl=wss://mosquitto:8083/mqtt`。
   - BackendProcessor：`replicas:1` 起步，投影 Worker 可按 consumer group 独立扩容，`terminationGracePeriodSeconds:45`，`podAntiAffinity`。
   - DeviceSimulator：`replicas:1`。
   - 健康探针指向对应路径，资源限制合理。
   - Pod Security Context：非 root、只读根文件系统、禁用特权模式。
6. `kubectl apply` 部署所有清单。

#### 验证标准
- 所有 Pod 进入 `Running`。
- Simulator Pod 内可通过 HTTP 和 MQTT 发送数据至网关，网关日志显示双协议接入。

- 滚动重启 Gateway 时，HTTP 请求不会返回假成功；关键 MQTT 消息可恢复投递。

### 第 8 步：端到端压测、故障演练与实验报告

3. **故障注入**：
   - 删除 Kafka Pod：验证 HTTP 返回 503，不产生假成功；MQTT 关键事件由 Broker 持久会话保护。

## 三、项目价值与结语

完成本计划后，你的简历可呈现：

> **IoTHunter —— 基于 .NET 10 的双协议智能手环数据网关**  
> - 设计并实现 **HTTP + MQTT-over-WebSocket 双协议接入网关**，以 Kafka ack 作为接入成功边界，避免内存队列假成功。  
> - 基于 Kafka、Polly、MQTT QoS 1 和持久会话实现普通遥测与关键事件的分级可靠性。  
> - 采用 Kafka 投影式 CQRS 架构，主事实库、Redis 最新值、时序视图独立消费，避免三库伪分布式事务。  
> - 使用 `event_id`、`sequence`、`recorded_at` 完成端到端幂等、旧值保护和故障重放。  
> - 采用 OpenTelemetry Propagators 传播 Trace Context 跨越 Kafka，实现跨协议、跨服务链路追踪。  
> - 使用 K3s + Cilium 部署，配置 NetworkPolicy、TLS、设备认证、MQTT ACL、Secret 与非 root 安全基线。

这份架构不再依赖“口号式零丢失”，而是明确每一段链路的可靠性边界：进入 Kafka 后由 Kafka 和幂等消费保证可恢复处理；进入 Kafka 前由 HTTP ack、MQTT QoS、Broker 持久化和设备端重试共同约束。这样才更接近 2026 年 .NET 微服务 IoT 平台的真实最佳实践。

## 主要修改条目

1. **基础技术栈升级**
   - 从 `.NET 8` 改为 `.NET 10 LTS`。
   - PostgreSQL/Npgsql 升级为 `PostgreSQL 16/17 + Npgsql 10`。
   - 时序存储从强制 `InfluxDB 2.7` 改为优先 `TimescaleDB`，可选 `InfluxDB/ClickHouse` 作为投影。

2. **修正 Gateway 可靠性边界**
   - 删除“写入内存 Channel 后立即返回 202”的设计。
   - 改为 **HTTP 只有 Kafka produce ack 成功后才返回 `202 Accepted`**。
   - Kafka 不可用、超时或熔断时返回 `503`。
   - 明确 `Channel` 只做背压缓冲，不承担可靠存储。

3. **修正 MQTT 可靠性设计**
   - 普通遥测允许 QoS 0 或 QoS 1。
   - 关键事件必须 QoS 1。
   - 禁止用 `CleanSession=true` 宣称零丢失。
   - 增加稳定 ClientId、持久会话、Broker 持久化、TLS、ACL 设计。

4. **重构 BackendProcessor 一致性模型**
   - 原“三存储全部成功后提交 Offset”改为 Kafka 投影式 CQRS。
   - 拆成：
     - `TelemetryPersistenceWorker`
     - `LatestValueProjectionWorker`
     - `TimeseriesProjectionWorker`
   - 每个 Worker 独立 consumer group、独立 offset、独立失败恢复。
   - 避免 PG/Redis/InfluxDB 伪分布式事务。

5. **补齐幂等与事件模型**
   - 增加 `event_id`、`sequence`、`recorded_at`、`received_at`、`schema_version`。
   - PG 唯一约束从 `(device_id, timestamp)` 改为 `event_id`。
   - Redis 更新 latest 前必须比较 `recorded_at` 或 `sequence`，防止旧消息覆盖新值。

6. **补齐安全基线**
   - Mosquitto 从 `allow_anonymous true` 改为 `allow_anonymous false`。
   - 增加 password file、ACL file、TLS、K8s Secret。
   - 增加 HTTP 设备鉴权、限流、payload schema 校验。
   - K8s 增加非 root、只读根文件系统、禁用特权模式。

7. **修正压测与验收口径**
   - P99 改为以 Kafka ack 完成时间计算，不再只计算写入内存队列时间。
   - 故障演练增加 Kafka、Gateway、Redis、时序投影存储故障场景。
   - 验收标准改为：普通遥测可靠性边界清晰，关键事件不产生假成功，投影可恢复追平。

整体上，文档现在不再依赖“口号式零丢失”，而是明确了进入 Kafka 前后的可靠性边界，更符合 2026 年 `.NET 10` 微服务 IoT 平台的最佳实践。
