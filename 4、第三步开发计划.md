## 3.1 子步骤一：Kafka 消费者骨架与多消费者组注册
**目标**：搭建三个独立的 Kafka 消费者 HostedService，每组使用独立的 Consumer Group。
### 3.1.1 动作项
1. **新建 `KafkaConsumerOptions`** (`Infrastructure/Options/KafkaConsumerOptions.cs`)：
   ```csharp
   namespace BackendProcessor.Infrastructure.Options;
   public class KafkaConsumerOptions {
       public string BootstrapServers { get; set; } = string.Empty;
       public string GroupId { get; set; } = string.Empty;
       public List<string> Topics { get; set; } = new();
   }
   ```
2. **更新 `appsettings.json`**：
   ```json
   "Kafka": {
     "BootstrapServers": "localhost:9092",
     "Consumers": {
       "Persistence": { "GroupId": "iot-persistence", "Topics": [ "telemetry.raw", "event.critical" ] },
       "LatestProjection": { "GroupId": "iot-latest-projection", "Topics": [ "telemetry.raw", "event.critical" ] },
       "TimeseriesProjection": { "GroupId": "iot-timeseries-projection", "Topics": [ "telemetry.raw", "event.critical" ] }
     }
   }
   ```
3. **实现 `KafkaConsumerFactory`** (`Infrastructure/Kafka/KafkaConsumerFactory.cs`)：
   - 封装 `ConsumerBuilder<Null, string>`。
   - **强制配置**：`EnableAutoCommit = false` (ADR-003)，`AutoOffsetReset = AutoOffsetReset.Earliest`。
   - **云原生背压配置**：设置 `MaxPollRecords = 500`。当系统处理慢时，Consumer 自动阻塞在 Broker 端，触发 Rebalance，交由 K8s HPA 扩容，**禁止在应用层使用 Channel 做动态背压**。
4. **创建三个 `BackgroundService` 骨架**：
   - `TelemetryPersistenceWorker` (group = `iot-persistence`)
   - `LatestValueProjectionWorker` (group = `iot-latest-projection`)
   - `TimeseriesProjectionWorker` (group = `iot-timeseries-projection`)
   - 每个 Worker 的 `ExecuteAsync` 中：提取 Kafka Header 的 `traceparent`，调用 `Propagators.DefaultTextMapPropagator.Extract` 恢复父 Span。
### 3.1.2 验证标准
- `dotnet build` 零错误。
- 启动 `BackendProcessor`，日志显示三个 Consumer 分配分区，无 AutoCommit 报警。

## 3.2 子步骤二：主事实库 Worker 与绝对幂等写入（核心修复）
**目标**：将 Kafka 消息批量写入 PG，以 `event_id` 为幂等键，**使用 Kafka 原生状态机管理 Offset，禁止手写内存偏移量**。
### 3.2.1 动作项
1. **创建 PostgreSQL 表结构** (可直接在 PG 中执行)：
   ```sql
   CREATE TABLE IF NOT EXISTS telemetry_records (
       event_id TEXT PRIMARY KEY,
       device_id TEXT NOT NULL,
       sequence BIGINT NOT NULL,
       metric_type TEXT NOT NULL,
       payload_json JSONB NOT NULL,
       recorded_at TIMESTAMPTZ NOT NULL,
       received_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
       schema_version INTEGER NOT NULL DEFAULT 1,
       reliability SMALLINT NOT NULL DEFAULT 0,
       created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
   );
   CREATE INDEX idx_device_time ON telemetry_records(device_id, recorded_at DESC);
   ```
2. **实现批处理逻辑** (在 `TelemetryPersistenceWorker` 内部)：
   - 累积 100 条或 5 秒窗口。
   - 构造批量 SQL：`INSERT INTO ... VALUES (...) ON CONFLICT (event_id) DO NOTHING;` (ADR-018)。
   - 写入失败：**抛出异常，不提交 Offset**，让 Kafka 重投，由 DB 唯一约束安全去重。
3. **🚨 终极修复：Offset 提交策略**：
   - **绝对禁止**维护 `lastCommittedOffset` 内存变量。
   - 每批写入 PG 成功后，遍历这批消息对应的 `ConsumeResult`，调用：
     ```csharp
     consumer.StoreOffset(result); // 告诉 Kafka 客户端：这个 offset 我处理完了
     ```
   - 在消费循环外或定时器中（如每 5 秒），调用 `consumer.Commit()` 将状态同步到 Broker。
   - 在 `StopAsync` 中，直接调用 `consumer.Close()`，Kafka 客户端会自动触发最后一次 Commit。
4. **指标**：`persistence_processed_total`、`persistence_batch_duration_ms`。
### 3.2.2 验证标准
- 发送 10 条，PG 有 10 条。发送相同 EventId 5 次，PG 仍是 10 条。
- 停止 PG，Processor 报错但不崩溃；重启 PG，数据自动追平，无重复死循环。

## 3.3 子步骤三：Redis 最新值投影 Worker
**目标**：更新 Redis，使用 Lua 脚本保证并发下旧消息不覆盖新消息。
### 3.3.1 动作项
1. **Redis 连接**：注册 `ConnectionMultiplexer` 为单例，配置 `abortConnect=false`。
2. **Lua 原子更新脚本** (在 `LatestValueProjectionWorker` 中硬编码)：
   ```lua
   local currentSeq = redis.call('HGET', KEYS[1], 'sequence')
   if not currentSeq or tonumber(ARGV[1]) >= tonumber(currentSeq) then
       redis.call('HSET', KEYS[1], 'metric_type', ARGV[2], 'payload_json', ARGV[3], 'recorded_at', ARGV[4], 'sequence', ARGV[1], 'reliability', ARGV[5])
       redis.call('EXPIRE', KEYS[1], 86400)
       return 1
   end
   return 0
   ```
3. **独立 Offset 提交**：Redis 写入失败只记日志，正常提交自身 Offset（不阻塞主事实库）。
### 3.3.2 验证标准
- 发送 seq 1→5，Redis 中 sequence 为 5。
- 发送 seq=3 旧消息，Redis 中仍为 5，指标 `stale_skipped_total` +1。
- 停止 Redis，Processor 不崩溃。

## 3.5 子步骤五：死信队列（DLQ）与异常处理
**目标**：非法数据写入死信 Topic，不污染主流程。
### 3.5.1 动作项
1. **DLQ 判定**：在反序列化后立即执行。
   - JSON 失败 → `reason="deserialization_failed"`
   - `device_id` 不匹配正则 `^[a-zA-Z0-9_-]+$` → `reason="invalid_device_id"`
   - 时间戳超出未来 1 小时或过去 7 天 → `reason="timestamp_out_of_range"`
2. **DLQ Producer**：新建独立 Producer (`ClientId = "iot-dlq-producer"`)，写入 `telemetry.deadletter`，Header 携带原消息和 `failure_reason`。
3. **指标**：`dlq_messages_total{reason="..."}`。

## 3.6 子步骤六：CQRS 查询 API（核心修复：杜绝脏读降级）
**目标**：提供严格边界的只读查询，**宁可返回错误，绝不提供过期数据**。
### 3.6.1 动作项
新建 `Controllers/QueriesController.cs` (路由 `api/v2/devices`)：
1. **`GET /{deviceId}/latest` (严格 Redis 边界)**：
   - 仅从 Redis 查询 `device:latest:{deviceId}`。
   - 如果存在，返回 200，附加 Header `X-Data-Freshness: cache_hit`。
   - 🚨 **终极修复**：如果 Redis 不存在，**直接返回 `404 Not Found`**。禁止降级查询 PG。保证拿到的一定是最新投影，避免脏读引发业务误判。
2. **`GET /{deviceId}/history` (PG 分页)**：
   - 从 `telemetry_records` 查询，限制 `pageSize <= 200`。
   - 返回 `page`, `totalCount`, `hasNextPage`。
3. **`GET /{deviceId}/timeseries` (时序聚合)**：
   - 使用 TimescaleDB 的 `time_bucket()` 聚合。
   - 若时序投影未启用，返回 `501 Not Implemented`。
### 3.6.2 验证标准
- 调用 `/latest` 有数据返回 200，无数据返回 404（绝不能返回旧历史数据）。
- 分页查询边界正确，最大 200 条限制生效。
