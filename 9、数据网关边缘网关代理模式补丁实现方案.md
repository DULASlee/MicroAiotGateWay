# 数据网关边缘网关代理模式补丁实现方案

> **版本**：V1.0  
> **适用范围**：第四步（DeviceSimulator 压测工具）和第七步（可观测性部署）  
> **设计原则**：网关本身不感知代理存在。Simulator 模拟真实设备通过边缘代理接入，代理负责 TLS 终结与公网脏流量清洗，向内网网关转发干净请求。


## 一、架构原理

```
真实设备 (公网) → 边缘代理 (Traefik/Nginx) → IoTGateway (内网)
                                       ↓
                              Prometheus 抓取代理指标
```

- **边缘代理职责**：TLS 终结、限流、WAF、公网脏流量过滤。
- **网关职责**：只接收内网干净流量，不做任何公网安全处理。
- **Simulator 职责**：可切换直连网关模式（本地开发）和代理模式（模拟真实公网设备）。


## 二、第四步补丁：Simulator 代理模式支持

### 2.1 修改 `DeviceSimulator/Infrastructure/SimulatorOptions.cs`

增加一个可选的代理 URL 配置项：

```csharp
public sealed class SimulatorOptions
{
    // ... 现有属性保持不变 ...

    /// <summary>
    /// 边缘代理 URL，模拟真实设备通过公网代理接入。
    /// 为空时直连网关（默认开发模式）。
    /// HTTP 示例: http://edge-proxy:8080
    /// MQTT 示例: edge-proxy  (配合 MqttPort)
    /// </summary>
    public string ProxyUrl { get; set; } = "";
}
```

### 2.2 修改 `DeviceSimulator/appsettings.json`

增加默认空配置：

```json
{
  "Simulator": {
    // ... 现有配置保持不变 ...
    "ProxyUrl": ""
  }
}
```

### 2.3 修改 `DeviceSimulator/Program.cs`

在注册 `HttpClient` 时，根据 `ProxyUrl` 决定 BaseAddress：

```csharp
.ConfigureServices((context, services) =>
{
    services.Configure<SimulatorOptions>(context.Configuration.GetSection("Simulator"));

    var proxyUrl = context.Configuration["Simulator:ProxyUrl"];
    var defaultBase = context.Configuration["Simulator:HttpBaseUrl"] ?? "http://localhost:5080";
    var effectiveBase = string.IsNullOrWhiteSpace(proxyUrl) ? defaultBase : proxyUrl;

    services.AddHttpClient("Gateway", client =>
    {
        client.BaseAddress = new Uri(effectiveBase);
        client.Timeout = TimeSpan.FromSeconds(10);
    });

    // ... 其余不变 ...
})
```

**逻辑**：若 `ProxyUrl` 非空，HTTP 请求发往代理；否则直连网关。

### 2.4 修改 `DeviceSimulator/Infrastructure/MqttTelemetrySender.cs`

在 `EnsureConnectedAsync` 中，`broker` 参数取值逻辑同样走 `ProxyUrl`：

```csharp
// 调用方在 SimulatorWorker 中根据 SimulatorOptions.ProxyUrl 决定 broker 地址
var broker = string.IsNullOrWhiteSpace(_options.ProxyUrl)
    ? _options.MqttBroker
    : _options.ProxyUrl;
```

### 2.5 验证标准

- 默认配置（`ProxyUrl=""`）下，Simulator 直连 Gateway，行为不变。
- 设置 `ProxyUrl=http://localhost:9999`（指向一个不存在的代理），Simulator 发送失败，日志显示连接拒绝——证明代理模式生效。
- 后续部署代理容器后，再将 `ProxyUrl` 指向真实代理地址，发送恢复正常。


## 三、第七步补丁：边缘代理 ServiceMonitor

### 3.1 新建 `deploy/observability/service-monitor-edge-proxy.yaml`

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: edge-gateway-proxy
  namespace: iothunter
spec:
  selector:
    matchLabels:
      app: edge-proxy
  endpoints:
    - port: metrics
      interval: 15s
      path: /metrics
```

### 3.2 部署时机

- 在第七步中，与 `deploy/observability/service-monitors.yaml`（IoTGateway、BackendProcessor、DeviceSimulator 的 ServiceMonitor）一同执行：
  ```bash
  kubectl apply -f deploy/observability/
  ```
- 如果集群中尚未部署边缘代理，此 ServiceMonitor 不会匹配到任何 Pod，对系统零影响。待边缘代理部署并打上 `app: edge-proxy` 标签后，Prometheus 自动开始抓取。

### 3.3 验证标准

- 部署边缘代理后，使用 `kubectl port-forward` 访问 Prometheus UI → Status → Targets，确认 `edge-gateway-proxy` 目标为 UP。
- Grafana 中可查看代理层指标（连接数、请求延迟、TLS 握手耗时等）。


## 四、风险规避

1. **Simulator 向后兼容**：`ProxyUrl` 默认为空，不影响任何现有测试脚本和本地开发流程。
2. **网关零修改**：IoTGateway 完全不感知请求来自直连设备还是边缘代理。代理在 HTTP Header 中添加的 `X-Forwarded-For` 等字段，网关可选择记录到日志，但不作为核心逻辑依赖。
3. **代理未就绪时无影响**：ServiceMonitor 匹配不到 Pod 时安静无动作，不会产生告警或错误。
4. **安全性**：生产环境代理必须启用 TLS（WSS/HTTPS），证书由 Cert-Manager 管理，代理与网关之间走内网明文或内部 TLS（视安全策略）。
