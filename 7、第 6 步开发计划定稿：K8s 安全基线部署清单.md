**目标**：部署明确的 NetworkPolicy，确保仅授权 Pod 可访问 Kafka、PG、Redis、Mosquitto。

### 6.3.1 动作项

1. **新建 `deploy/network-policy.yaml`**

   - `allow-gateway-to-kafka`：IoTGateway → kafka:9092。
   - `allow-gateway-to-mosquitto`：IoTGateway → mosquitto:1883, 8083。
   - `allow-processor-to-kafka`：BackendProcessor → kafka:9092。
   - `allow-processor-to-pg`：BackendProcessor → postgres:5432。
   - `allow-processor-to-redis`：BackendProcessor → redis:6379。
   - `allow-sim-to-gateway`：DeviceSimulator → IoTGateway:80。
   - `allow-sim-to-mosquitto`：DeviceSimulator → mosquitto:1883, 8083。
   - 默认：该命名空间内 Pod 之间的非授权流量将被拒绝（`podSelector: {}` + `ingress/egress` 空规则作为默认拒绝）。
2. **执行**

   ```bash
   kubectl apply -f deploy/network-policy.yaml
   ```

### 6.3.2 验证标准

- 从调试 Pod 内 `curl http://iot-gateway:80/health/ready` 应被拒绝（超时或连接拒绝）。

## 6.4 子步骤四：生成 TLS 证书并以 Secret 形式挂载到 Gateway 和 Mosquitto

**目标**：生成自签 CA、Gateway 服务端证书、Mosquitto 服务端证书，在 IoTGateway 和 BackendProcessor 启用 HTTPS。

### 6.4.1 动作项

1. **生成自签 CA 和服务证书**

   ```bash
   # 生成 CA
   openssl req -x509 -newkey rsa:4096 -days 365 -nodes -keyout ca.key -out ca.crt -subj "/CN=IoTHunter CA"

   # 生成 Gateway 证书
   openssl req -newkey rsa:2048 -nodes -keyout gateway.key -out gateway.csr -subj "/CN=iot-gateway.iothunter.svc"
   openssl x509 -req -in gateway.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out gateway.crt -days 365

   # 生成 Mosquitto 证书（同理，CN=mosquitto.iothunter.svc）
   openssl req -newkey rsa:2048 -nodes -keyout mosquitto.key -out mosquitto.csr -subj "/CN=mosquitto.iothunter.svc"
   openssl x509 -req -in mosquitto.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out mosquitto.crt -days 365

   # 创建 Secret
   kubectl create secret tls gateway-tls --cert=gateway.crt --key=gateway.key -n iothunter
   kubectl create secret tls mosquitto-tls --cert=mosquitto.crt --key=mosquitto.key -n iothunter
   ```

   > **证书作用域说明**（清言审查补充）：本步生成的 TLS 证书仅用于 K8s 集群内部 Pod 间通信，CN 对齐内部 Service DNS 名称（`.svc`）。若未来需支持外部设备直连 Mosquitto WSS，TLS 终止应由 Ingress Controller 统一处理（如 Nginx Ingress + Cert-Manager），证书签发与续期由 Cert-Manager 管理，不在本步范围内。

2. **更新 Gateway Deployment**
   - 挂载 `gateway-tls` Secret 到 `/app/certs`。
   - 环境变量 `ASPNETCORE_Kestrel__Certificates__Default__Path=/app/certs/tls.crt`。
   - 暴露容器端口 443。

3. **更新 Mosquitto Deployment**
   - 挂载 `mosquitto-tls` Secret 到 `/mosquitto/config/certs/`。
   - 修改 `mosquitto.conf` 中 TLS listener 配置。

### 6.4.2 验证标准

- `curl -k https://iot-gateway.iothunter.svc/health/ready` 返回 Healthy。
- MQTT 客户端使用 TLS 连接 `mosquitto.iothunter.svc:8883` 成功。


## 6.5 子步骤五：Deployment 与 Service 编写（IoTGateway + BackendProcessor + DeviceSimulator + Mosquitto）

**目标**：为四个核心服务编写完整的 Deployment 和 Service 清单，包含健康探针、资源限制、优雅停机参数、Pod 反亲和性。

### 6.5.1 动作项

1. **新建 `deploy/iot-gateway.yaml`**
   - Deployment `replicas:2`。
   - `ports`：80, 9464（Prometheus metrics 端口按 ADR-013 声明）。
   - `livenessProbe`：HTTP GET `/health/live`（若未定义则指向 `/health/ready`），`initialDelaySeconds:10`, `periodSeconds:10`。
   - `readinessProbe`：HTTP GET `/health/ready`，`initialDelaySeconds:5`, `periodSeconds:5`。
   - 环境变量从 Secret 注入：`KAFKA_BOOTSTRAP_SERVERS`、`MQTT_BROKER_URL=ws://mosquitto:8083/mqtt`。
   - `resources.limits`：CPU 1.0, Memory 256Mi。
   - `terminationGracePeriodSeconds:30`。
   - `podAntiAffinity: preferredDuringSchedulingIgnoredDuringExecution`，按 `app=iot-gateway` 分散。
   - 挂载 `gateway-tls` Secret 到 `/app/certs`。
   - Service `iot-gateway` ClusterIP，暴露 80、9464。

2. **新建 `deploy/backend-processor.yaml`**
   - Deployment `replicas:1`（投影 Workers 可按 consumer group 扩容至 3）。
   - `ports`：80, 9464。
   - `livenessProbe`、`readinessProbe` 同上。
   - 环境变量从 Secret 注入：`KAFKA_BOOTSTRAP_SERVERS`、`POSTGRES_CONNECTION_STRING`、`REDIS_CONNECTION_STRING`。
   - `resources.limits`：CPU 2.0, Memory 512Mi（批处理开销）。
   - `terminationGracePeriodSeconds:45`（消费者 Offset 提交 + 缓冲区刷出 + `consumer.Close()` 时间）。
   - Pod 反亲和性。
   - Service `backend-processor` ClusterIP，暴露 80, 9464。

3. **新建 `deploy/device-simulator.yaml`**
   - Deployment `replicas:1`（按需扩容）。
   - `ports`：9464。
   - 环境变量：`SIMULATOR__PROTOCOL=http`、`SIMULATOR__HTTPBASEURL=http://iot-gateway`。
   - `resources.limits`：CPU 1.0, Memory 128Mi。

4. **新建 `deploy/mosquitto.yaml`**
   - Deployment `replicas:1`。
   - `ports`：1883 (MQTT), 8083 (WebSocket), 8883 (MQTT TLS，按需)。
   - 挂载 `mosquitto-config` Secret 到 `/mosquitto/config/`。
   - 挂载 `mosquitto-tls` Secret 到 `/mosquitto/config/certs/`。
   - Service `mosquitto` ClusterIP，暴露 1883, 8083, 8883。

5. **执行**

   ```bash
   kubectl apply -f deploy/
   kubectl rollout status deployment -n iothunter
   ```

### 6.5.2 验证标准

- `kubectl get pods` 所有服务 Pod Running，READY 1/1。
- `kubectl port-forward service/iot-gateway 5080:80` 后 `curl http://localhost:5080/health/ready` 返回 Healthy。
- `kubectl logs -f deployment/iot-gateway` 显示 `MQTT connected and subscribed`。
- `kubectl logs -f deployment/backend-processor` 显示 Consumer Group 分配分区成功。
- `kubectl logs -f deployment/device-simulator` 输出发送成功日志。


## 6.6 子步骤六：Pod 安全上下文

**目标**：为每个 Deployment 添加 `securityContext`，禁止特权模式、限制 root 运行、设置只读根文件系统。

### 6.6.1 动作项

1. 在每个 Deployment 的 `spec.template.spec` 中添加：

   ```yaml
   securityContext:
     runAsNonRoot: true
     runAsUser: 1000
     fsGroup: 1000
     readOnlyRootFilesystem: true
   ```

2. 对于需要写入临时文件的容器（如 Mosquitto 数据目录），挂载 `emptyDir` 卷。

### 6.6.2 验证标准

- `kubectl exec -it deploy/iot-gateway -- whoami` 返回非 root 用户名（如 `app` 或 `1000`）。
- `kubectl exec -it deploy/iot-gateway -- touch /app/test` 应失败（read-only filesystem）。


## 6.7 子步骤七：滚动重启验证与优雅停机

**目标**：模拟 K8s 滚动更新，验证 Gateway 的 HTTP 请求不产生假成功，Processor 的 Consumer Offset 正确提交。

### 6.7.1 动作项

1. 启动持续发送脚本：

   ```bash
   kubectl exec -it deploy/device-simulator -- /bin/sh -c "while true; do curl -X POST http://iot-gateway/api/v1/telemetry -H 'Content-Type: application/json' -d '{\"deviceId\":\"dev-test\",\"metricType\":\"test\",\"timestamp\":$(date +%s)000,\"sequence\":1}'; sleep 0.1; done"
   ```

2. 触发滚动重启：

   ```bash
   kubectl rollout restart deployment/iot-gateway -n iothunter
   kubectl rollout status deployment/iot-gateway -n iothunter --watch
   ```

3. 观察 Simulator 日志：503 出现期间无假成功（`status=success`）返回。
4. 验证 Processor 的 Offset 是否在重启后 Lag 归零。

### 6.7.2 验证标准

- 滚动重启期间，Simulator 日志出现短暂 503，无任何 `status=success` 返回（重启窗口）。
- 重启完成后，Kafka Consumer Group `iot-persistence` 的 Lag 在 10 秒内降至 0。
- Gateway 新 Pod 日志中出现 `Kafka producer initialized` 后，请求恢复 202。


## 6.8 子步骤八：端到端安全基线验证脚本

**目标**：自动化验证 NetworkPolicy、TLS、Secret、探针和安全上下文，确认整套部署满足 ADR 要求。

### 6.8.1 动作项

1. **新建 `scripts/e2e-verify-step6.ps1`**

   核心流程：
   - 检查所有 Pod 是否在 `iothunter` 命名空间且 Running。
   - **NetworkPolicy 验证**：从非法 Pod（不同标签） `curl` Gateway 应超时/拒绝；从 Simulator `curl` Gateway 应成功。
   - **TLS 验证**：`curl -k https://iot-gateway.iothunter.svc/health/ready` 成功。
   - **Secret 验证**：Gateway Pod 内 `echo $KAFKA_BOOTSTRAP_SERVERS` 输出 `kafka:9092`。
   - **探针验证**：`kubectl describe pod` 检查 `Liveness` 和 `Readiness` 均为 OK。
   - **安全上下文验证**：Pod 内 `whoami` 输出非 root，`touch /app/test` 失败。
   - **优雅停机验证**：触发滚动重启，检查 Processor Lag 恢复为 0。
   - 发送一批测试数据，通过 BackendProcessor 查询 API 验证数据完整。

### 6.8.2 验证标准

- 脚本执行完毕，输出 `ALL STEP6 CHECKS PASSED`。
- 所有 ADR-008、ADR-013、ADR-017、ADR-021 的要求均通过验证。

## 小结

第六步是整个项目从“能写”到“能跑”的最后一公里。我们在这里完成了 K8s 命名空间隔离、Secret 收敛敏感信息、零信任网络策略、TLS 证书部署、四类服务的 Deployment 与探针配置，以及 Pod 安全上下文的加固。这一步完成后，我们的网关集群已经不是在开发机上裸奔的服务，而是一个可以被渗透测试、可以被滚动更新、可以被水平扩容的云原生系统。下一步，我们将为它装上全栈可观测性的眼睛，让 Prometheus 和 Jaeger 接管每一寸链路。
